

## Day 2｜文本 / Markdown 导入为 JSONL

**今日目标：**  
==把一批原始文本或 Markdown 文件导入为统一的 JSONL 格式，形成后续清洗、切块、embedding、RAG 检索评测的第一批原始语料。今天不追求复杂清洗，重点是建立“原始材料 → 结构化样本”的第一条数据管线。==

### 你主导

1. 先定义单条样本的 ==schema==。建议至少包含：
    

```json
{
  "id": "doc_0001",
  "source_path": "data/raw/example.md",
  "title": "example",
  "text": "正文内容",
  "metadata": {
    "file_type": "markdown"
  }
}
```

2. 自己准备 20 条样本文档。可以来自你的学习笔记、README、文章片段、课程笔记或 AI-wiki 中适合做测试的小段内容。今天先保证材料可控、来源清楚。

`我选择D:\obsidian\备考mock下的学习笔记`

    
3. 写下验收标准：  
    “导入脚本运行后，是否能稳定输出 20 行 JSONL？每一行是否都是合法 JSON？是否保留了原始文件路径？”
    

### Codex 主导

让 Codex 在你给定 schema 后做一个小 patch：

- 新增一个 `scripts/import_markdown.py`
    
- 读取 `data/raw/` 下的 `.md` 或 `.txt`
    
- 输出到 `data/processed/documents.jsonl`
    
- 每个文件生成一条 JSONL 记录
    
- 不做复杂清洗，只做基本读取、空文件跳过、UTF-8 处理
    

可直接复制给 Codex 的指令：

> 请先说明你的实施计划、准备修改哪些文件、每个文件的作用、风险和验证方式。得到确认后，只做一个小 patch：新增一个 Markdown/TXT 导入脚本，从 `data/raw/` 读取文件，并输出 `data/processed/documents.jsonl`。每条记录包含 `id`、`source_path`、`title`、`text`、`metadata.file_type`。不要加入切块、embedding 或 RAG 功能。

### AI-wiki 取材

`directions/data.md`

**今天拆用的可做项目：**  
AI-wiki 的数据方向建议“为自己的笔记/文章构建 RAG 数据集”，其中第一步就是把 Markdown 等原始材料整理成可处理的数据格式；后续才进入清洗、切块、embedding 和检索评测。

**今天对应的检查点：**  
今天主要对应两个检查点：第一，能解释一个样本从原始数据到训练或检索输入的全过程；第二，能写脚本统计或处理基础数据状态。今天先完成“原始数据 → 结构化 JSONL”的最小管线。

**必要补充：**  
AI-wiki 没有给出具体导入脚本实现，所以今天只补充 Python 标准库知识：`pathlib`、`json`、文件读写、UTF-8 编码。

### 共同复盘／与 Codex 讨论

1. 为什么今天先输出 JSONL，而不是直接存进 SQLite、向量库或 Chroma？  
    你先写自己的判断，再让 Codex 从“可调试性、可移植性、后续扩展”三个角度反驳或补充。
    
2. 每个文件一条记录是否足够？  
    让 Codex 提出另一种 schema，例如“按段落一条记录”或“按标题层级一条记录”，并比较它们对后续 RAG 切块的影响。
    

这个半辅助拆分会继续沿用“你负责规格、判断与验收，Codex 负责小 patch、测试和排错”的模式。
